<objective>
Wire tool mapping and control protocol into the existing event bridge, provider, and prompt builder. This is where Phase 2 comes alive -- the event bridge handles tool_use content blocks and emits toolcall events, the provider routes control_request messages through the control handler, and the prompt builder applies reverse name/arg mapping when replaying tool history.

Purpose: After this plan, the full "Claude proposes, pi executes" loop works: Claude proposes a tool, our extension denies execution via control protocol, maps the tool name and arguments to pi's format, and emits toolcall events. Pi then executes the tool natively and the result appears in the next conversation turn with correct Claude-side naming.

Output: Modified event-bridge.ts, provider.ts, prompt-builder.ts with updated tests.
</objective>

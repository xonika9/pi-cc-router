<objective>
Create the schema-only MCP server, config generation, break-early subprocess control, and simplified control handler for the new break-early architecture.

Purpose: The previous in-process HTTP MCP architecture was proven fundamentally flawed (MCP tools bypass control protocol, no executeTool API in pi, HTTP transport complexity). Empirical testing revealed a much simpler architecture: use `--permission-prompt-tool stdio` to enable control_request for MCP tools, kill the subprocess at `message_stop` before Claude auto-executes tools (break-early pattern), and use a minimal raw JSON-RPC stdio MCP server that only implements `tools/list` (tools/call is never reached because the subprocess is killed first).

Output: `src/mcp-schema-server.cjs` (schema-only MCP server), `src/mcp-config.ts` (tool discovery + config generation), updated `process-manager.ts` (CLI flag changes), simplified `control-handler.ts`, `provider.ts` with break-early logic.
</objective>

1. Create `src/mcp-schema-server.cjs` — a minimal CommonJS script (~30 lines) that Claude CLI spawns as an MCP server process:

   ```javascript
   #!/usr/bin/env node
   // Schema-only MCP server. Reads tool schemas from a JSON file.
   // Only implements initialize + tools/list. tools/call is never reached
   // because the parent process kills the Claude subprocess at message_stop
   // before tool execution (break-early pattern).
   "use strict";
   const fs = require("fs");
   const readline = require("readline");

   const schemaPath = process.argv[2];
   if (!schemaPath) { process.exit(1); }

   let tools = [];
   try { tools = JSON.parse(fs.readFileSync(schemaPath, "utf-8")); } catch { process.exit(1); }

   const rl = readline.createInterface({ input: process.stdin });
   rl.on("line", (line) => {
     let msg;
     try { msg = JSON.parse(line); } catch { return; }

     if (msg.method === "initialize") {
       const resp = { jsonrpc: "2.0", id: msg.id, result: {
         protocolVersion: "2024-11-05",
         capabilities: { tools: {} },
         serverInfo: { name: "custom-tools", version: "1.0.0" }
       }};
       process.stdout.write(JSON.stringify(resp) + "\n");
     } else if (msg.method === "tools/list") {
       const resp = { jsonrpc: "2.0", id: msg.id, result: { tools } };
       process.stdout.write(JSON.stringify(resp) + "\n");
     }
     // notifications/initialized: no response needed (notification)
     // tools/call: never reached (break-early kills subprocess first)
   });
   ```

   Key constraints:
   - Must be `.cjs` (CommonJS) because Claude CLI spawns it with `node`
   - No SDK dependency — raw JSON-RPC over stdio
   - No `console.log` — only `console.error` for debugging
   - Reads tool schemas from temp file passed as argv[2]

   export function handleControlRequest(
     msg: ClaudeControlRequest,
     stdin: NodeJS.WritableStream,
   ): boolean {
     // Guard against malformed control_request messages
     if (!msg.request_id || !msg.request) {
       console.error(
         "[pi-claude-cli] Malformed control_request: missing request_id or request object",
         msg,
       );
       return false;

     // path where all tool_use blocks are user-side MCP tools. We want Claude to
     // execute those via their MCP servers.

       request_id: msg.request_id,
       response: {

         response: { behavior: "allow" },
       },
     };

     return true;

   Add tracking state inside the async IIFE, before the readline setup:

   In the `rl.on("line")` callback, add a `broken` guard at the very top of the callback to prevent buffered readline lines from firing after `rl.close()`. This is critical because Node's readline can buffer lines before `rl.close()` takes effect, causing race conditions with `stream.end()`:
   ```typescript
   rl.on("line", (line: string) => {
     if (broken) return; // Guard: ignore buffered lines after break-early

     const msg = parseLine(line);
     if (!msg) return;

     if (msg.type === "stream_event") {
       bridge.handleEvent(msg.event);

       // Track tool_use blocks for break-early decision
       if (msg.event.type === "content_block_start" && msg.event.content_block?.type === "tool_use") {
         const toolName = msg.event.content_block.name;
         sawAnyTool = true;
         if (!toolName.startsWith(MCP_PREFIX) || toolName.startsWith(CUSTOM_TOOLS_MCP_PREFIX)) {
           // Built-in tool (no mcp__ prefix) OR custom tool (mcp__custom-tools__*)
           sawBuiltInOrCustomTool = true;
         }
       }

       // Break-early at message_stop: kill subprocess before CLI auto-executes tools
       if (msg.event.type === "message_stop" && sawBuiltInOrCustomTool) {
         broken = true; // Set guard BEFORE rl.close() to prevent buffered lines
         // Pi will execute these tools. Kill subprocess to prevent CLI from executing them.
         if (proc && !proc.killed) {
           proc.kill("SIGKILL");
         }
         rl.close();
         return; // Don't process further — done event already pushed by event bridge
       }
     } else if (msg.type === "result") {
       // ...existing result handling...
     } else if (msg.type === "control_request") {
       handleControlRequest(msg, proc!.stdin!);
     }
   });
   ```

   IMPORTANT: The event bridge's `handleMessageStop` already pushes the "done" event and calls `stream.end()`. The break-early logic just needs to set the broken guard, kill the process, and close readline. It does NOT need to push done/end again. The `broken` flag MUST be set BEFORE calling `rl.close()` to prevent any buffered lines from racing with the stream end.

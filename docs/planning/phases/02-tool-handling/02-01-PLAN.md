  <action>
    **First, update `src/types.ts`:**

    Replace the existing `ClaudeControlRequest` interface with the verified wire protocol format:
    ```typescript
    export interface ClaudeControlRequest {
      type: "control_request";
      request_id: string;
      request: {
        subtype: "can_use_tool";
        tool_name: string;
        input: Record<string, unknown>;
      };
    }
    ```
    This is a breaking change from the placeholder definition in Phase 1. The `request_id` field and nested `request` wrapper are required by the wire protocol. Remove the old flat `subtype`, `tool_name`, `tool_input` fields. Also update the `NdjsonMessage` union type -- no changes needed there since the type name is the same.

    **Write tests** in `tests/control-handler.test.ts` covering all behaviors above. Use a mock writable stream for `proc.stdin` to capture what gets written. Run tests -- they must FAIL (RED).

    Then create `src/control-handler.ts`:

    1. Export constant `TOOL_EXECUTION_DENIED_MESSAGE = "Tool execution is unavailable in this environment."` (matches reference project).

    2. Export constant `MCP_PREFIX = "mcp__"`.

    3. Define internal `ControlResponse` interface:
       ```typescript
       interface ControlResponse {
         type: "control_response";
         request_id: string;
         response: {
           subtype: "success";
           response: {
             behavior: "allow" | "deny";
             message?: string;
           };
         };
       }
       ```

    4. Export function `handleControlRequest(msg: ClaudeControlRequest, stdin: NodeJS.WritableStream): boolean`:
       - Extract `tool_name` from `msg.request.tool_name`
       - Check `tool_name.startsWith(MCP_PREFIX)` to determine allow vs deny
       - Build `ControlResponse` object with matching `msg.request_id`
       - For deny: include `message: TOOL_EXECUTION_DENIED_MESSAGE`
       - For allow: omit `message` field
       - Write `JSON.stringify(response) + "\n"` to stdin
       - Return `true` if allowed, `false` if denied

    Accept `stdin` as `NodeJS.WritableStream` (not `ChildProcess`) for easier testing and decoupling. The caller (provider.ts in Plan 02) will pass `proc.stdin!`.

    Handle malformed control_request (missing request_id or request object) by logging to console.error and returning false (per Claude's discretion in CONTEXT.md).

    Run tests -- they must PASS (GREEN).

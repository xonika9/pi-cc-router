⏺ Now I have a thorough understanding of the entire codebase. Let me plan and
  implement the persistent subprocess sessions feature.

  The key insight from my research: The Claude CLI with --input-format 
  stream-json already supports multi-turn — you can write multiple NDJSON user
  messages to stdin. But currently the code uses --no-session-persistence and
  kills the subprocess after each turn.

  The best approach for this codebase is to use --resume <session_id> to resume
  sessions across subprocess restarts. This is simpler than keeping a single
  subprocess alive (which would conflict with the break-early pattern that kills
   the process). The result message already returns a session_id — we just need
  to capture it and pass --resume on subsequent calls.

  This eliminates the quadratic token growth (Claude CLI manages conversation
  history internally) and avoids replaying the full conversation as a flattened
  prompt.

  Implementation Summary: Persistent Subprocess Sessions (#1)

  Problem

  Each request spawned a fresh claude -p subprocess, replaying the entire
  conversation as a flattened text prompt. This caused:
  - ~12s startup overhead per request
  - Quadratic token growth as history accumulated
  - Redundant reprocessing of identical context

  Solution

  Use the Claude CLI's built-in --resume <session_id> flag to resume prior
  conversations. The CLI persists session state to disk, so subsequent requests
  only need to send the new user message — not replay the full history.

  Changes

  New file: src/session-manager.ts
  - In-memory Map<cwd, sessionId> tracking sessions per working directory
  - setSessionId() / getSessionId() / clearSessionId() / clearAllSessions()

  Modified: src/process-manager.ts
  - Removed --no-session-persistence flag (sessions now persist to disk)
  - Added resumeSessionId option to spawnClaude() — passes --resume <id> when
  set

  Modified: src/provider.ts
  - On first request: full prompt + system prompt (no --resume)
  - On subsequent requests: --resume <sessionId> + only the new user turn via
  buildResumePrompt()
  - Captures session_id from both system init and result messages
  - Clears session on errors (result error, subprocess crash, process error) so
  next request falls back to full prompt

  Modified: src/prompt-builder.ts
  - New buildResumePrompt() — extracts just the tool results + latest user
  message (the delta since last turn), avoiding redundant history replay
